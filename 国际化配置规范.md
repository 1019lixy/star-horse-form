# 组件国际化配置规范

## 背景

在组件开发和发布过程中，确保国际化内容正确显示是一个关键问题。本规范旨在解决因国际化语言包加载顺序导致的显示异常问题。

## 核心问题

国际化内容在模块导入时立即计算，而此时语言包可能尚未完全加载，导致显示异常。

## 配置规范

### 1. 避免模块级别的国际化计算

**错误做法：**
```typescript
// 错误：模块导入时立即计算国际化内容
export const config = {
  label: i18n("key"),
  // 其他配置...
};
```

**正确做法：**
```typescript
// 正确：使用函数形式延迟计算国际化内容
export const getConfig = () => ({
  label: i18n("key"),
  // 其他配置...
});
```

### 2. 使用函数形式导出配置

对于所有包含国际化内容的配置对象，应使用函数形式导出，确保在需要使用时才计算国际化内容。

**规范格式：**
```typescript
// 返回函数而不是直接的配置对象
export const getConfigName = () => ({
  // 配置内容，包含i18n调用
});

// 保留原有的导出名称，确保向后兼容
export const configName = {};
```

### 3. 确保语言包加载完成后再使用配置

在组件中使用配置时，确保在国际化语言包加载完成后再调用配置函数：

```typescript
// 示例：在组件中使用
import {getConfig} from "@/path/to/config";
import {loadLanguage, currentLang} from "@/lang";

// 确保在国际化加载完成后调用配置函数
async function init() {
  await loadLanguage(currentLang);
  const config = getConfig();
  // 使用配置...
}
```

### 4. 组件生命周期中的处理

在组件的生命周期钩子中，确保国际化加载完成后再初始化依赖国际化的配置：

```typescript
onMounted(async () => {
  await loadLanguage(currentLang);
  // 初始化配置
  config.value = getConfig();
});
```

### 5. 响应式监听语言变化

当语言发生变化时，需要重新计算国际化内容：

```typescript
import {onLanguageChange} from "@/lang";

// 注册语言变化监听
const unregister = onLanguageChange(() => {
  // 语言变化时重新获取配置
  config.value = getConfig();
});

// 组件卸载时取消监听
onUnmounted(() => {
  unregister();
});
```

## 兼容性处理

为了确保向后兼容性，对于修改现有的配置文件，应保留原有的导出名称：

```typescript
// 新的函数形式导出
export const getConfig = () => ({
  // 配置内容
});

// 保留原有的导出名称，提供空对象或默认实现
export const config = {};
```

## 适用范围

本规范适用于所有包含国际化内容的配置文件，包括但不限于：

1. 容器配置文件（containerConfig.ts）
2. 项目配置文件（itemsConfig.ts）
3. 布局配置文件（layouts.ts）
4. 其他任何包含i18n调用的配置对象

## 测试验证

修改完成后，需要验证以下场景：

1. 组件首次加载时国际化内容是否正确显示
2. 切换语言时国际化内容是否正确更新
3. 组件间交互时国际化内容是否一致
